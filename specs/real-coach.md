# PRD: Real Coach — 영상 분석 & 코칭 피드백

> Status: DRAFT
> Type: PRD (Product Requirements Document)
> Created: 2026-03-22
> Last Updated: 2026-03-22

## 1. 배경

### 현재 구현된 것

Atom은 집에서 혼자 복싱 훈련하는 사람을 위한 **오디오 가이드 드릴 앱**이다.

**현재 사용자 플로우:**
```
세션 설정 → 드릴 플랜 생성 → 오디오 재생 → 세션 종료 → 기록
(난이도/라운드)  (템플릿+셔플)     (실시간 큐)    (요약)      (통계)
```

**작동하는 기능:**
- 3단계 난이도, 다양한 주제별 60개 드릴 템플릿
- 가중치 기반 콤보 풀에서 랜덤 드릴 플랜 생성
- 한국어 음성 큐로 실시간 오디오 가이드 세션 (사전 녹음 청크 + TTS)
- 라운드별 MP3 조립 및 글자 단위 타임스탬프 동기화
- 세션 로깅 (완료 라운드, 시간, 전달된 세그먼트)
- 유저 프로필 집계 (총 세션 수, 총 훈련 시간)
- React Native/Expo 모바일 앱 (세션 설정 → 플랜 미리보기 → 실시간 드릴 → 세션 종료)

**현재 기술 스택:**
- FastAPI 백엔드 + SQLite (async SQLAlchemy)
- ElevenLabs / edge-tts (음성)
- LLMClient (Claude/GPT) — 사용 가능하지만 핵심 플로우에서 미사용
- React Native / Expo 모바일 앱

### 빠져 있는 것

앱은 **드릴 플레이어**이지, **코치**가 아니다. 사용자에게 *뭘 하라*고는 말하지만, *실제로 뭘 했는지*는 전혀 모른다. 피드백 루프가 제로다:

- 사용자가 드릴을 제대로 따라했는지 알 수 없음
- "세션 완료" 외에 성과 추적 없음
- 적응형 난이도 없음 — 매번 수동으로 레벨 선택
- 코칭 인사이트 없음 — "훅이 부족해요" 같은 피드백 불가
- 동기 부여 루프 없음 — 시간 경과에 따른 향상을 볼 수 없음

결과: 앱이 트레이너가 아니라 **메트로놈** 느낌이다.

## 2. 문제 정의

> 혼자 복싱 훈련하는 사람은 객관적인 피드백을 받을 방법이 없다. 체육관 코치는 시간당 5~10만원이고, 영상을 찍어서 나중에 보는 건 귀찮고 주관적이다. 사용자에게는 "드릴을 제대로 했는가?"와 "다음에 뭘 연습해야 하는가?"를 자동으로 알려주는 방법이 필요하다.

**핵심 인사이트:** 우리는 이미 *기대 펀치 시퀀스*를 가지고 있다 (드릴 플랜에서). 영상에서 *실제 펀치 시퀀스*를 추출할 수 있다면, 비교는 자명해지고 — 코칭 피드백이 가능해진다.

## 3. 타겟 유저

**주요 대상:** 집에서 혼자 훈련하는 복싱 입문~중급자
- 이미 Atom으로 오디오 가이드 드릴 사용 중
- 카메라 달린 스마트폰 보유
- 주 2~4회 훈련
- 실력 향상을 원하지만 정기 코칭을 받기엔 비용 부담

**현재 대상 아님:**
- 상급자/선수 (시퀀스 확인이 아니라 폼 분석이 필요)
- 그룹 수업 환경
- 세션 중 실시간 코칭

## 4. 제품 비전

Atom을 **오디오 드릴 플레이어**에서, 세션을 보고 놓친 걸 알려주고 약점 기반으로 다음 운동을 설계해주는 **AI 복싱 코치**로 전환한다.

**피봇 후 사용자 플로우:**
```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  세션       │    │  영상 촬영  │    │  피드백     │    │  다음 세션  │
│  시작       │───▶│  + 훈련     │───▶│  & 점수     │───▶│  추천       │
│  (설정)     │    │             │    │  확인       │    │             │
└─────────────┘    └─────────────┘    └─────────────┘    └──────┬──────┘
      ▲                                                         │
      └─────────────────────────────────────────────────────────┘
                          지속적 개선 루프
```

## 5. 성공 지표

| 지표 | 목표 | 측정 방법 |
|------|------|-----------|
| 분석 소요 시간 | 업로드 → 피드백 5분 이내 | 서버 타이머: 업로드 시각 → 결과 준비 시각 |
| 펀치 인식 정확도 | 클래스별 F1 > 0.7 (잽, 크로스, 훅, 어퍼컷, 바디샷) | 수동 어노테이션 10개 테스트 영상 대비 검증 |
| 피드백 구체성 | 특정 콤보와 세그먼트를 언급 (뭉뚱그린 칭찬 X) | 정성 평가: 어느 콤보에서 문제가 있었는지 피드백이 말하는가? |
| 추천 적응성 | 약한 펀치 타입을 타겟하는 다음 세션 추천 | 훅 정확도 3세션 연속 낮음 → 훅 중심 드릴 추천되는지 확인 |
| 유저 인게이지먼트 | 피드백 받는 유저의 세션 수 30%+ 증가 | 영상 분석 도입 전후 세션 빈도 비교 |

## 6. 기능 요구사항

### P0: 반드시 필요 (MVP)

**F1. 영상 업로드**
- 사용자가 훈련 세션을 휴대폰 카메라로 촬영
- 세션 종료 후 영상 업로드 (세션 중 아님)
- .mp4, .mov 지원, 최대 500MB
- 업로드 진행률 표시

**F2. 펀치 감지**
- 영상에서 5가지 펀치 인식: 잽, 크로스, 훅, 어퍼컷, 바디샷
- 프레임별 포즈 키포인트 추출 (MediaPipe)
- TCN(Temporal Convolutional Network)으로 펀치 분류
- 프레임 레벨 예측에서 개별 펀치 이벤트 추출
- 9분 영상 기준 5분 이내 처리 목표로 10fps 샘플링

**F3. 세그먼트별 비교**
- 드릴 플랜 세그먼트에 맞춰 영상을 시간 구간으로 분할
- *세그먼트별로* 감지된 펀치 vs 기대 펀치 비교
- 세그먼트별 점수 및 세션 전체 정확도 산출
- 방어 전용 세그먼트 처리 (채점에서 제외, 페널티 없음)
- 출력 예시: "세그먼트 2 (원투 훅): 기대 [잽, 크로스, 훅], 실제 [잽, 크로스] — 훅 누락"

**F4. 코칭 피드백**
- 비교 결과 기반으로 LLM이 한국어 피드백 2~3문장 생성
- 특정 콤보와 세그먼트를 언급해야 함 ("잘했어요" 식의 뭉뚱그림 X)
- 성적이 가장 나쁜 상위 3개 세그먼트 강조
- 분석당 LLM 1회 호출 (비용 통제)

**F5. 피드백 화면**
- 세션 점수 표시 (0~100%)
- 펀치별 정확도 분류 표시 (잽/크로스/훅/어퍼컷/바디샷)
- 세그먼트별 결과 표시 (어떤 콤보에서 문제 있었는지)
- LLM 코칭 메시지 한국어로 표시

### P1: 있으면 좋음

**F6. 다음 세션 추천**
- 규칙 기반 엔진이 최근 5세션의 펀치별 정확도 추적
- 레벨 조정: 3세션 연속 >80% → 레벨 업 제안, <50% → 레벨 다운
- 포커스 선택: 가장 약한 펀치 타입 → 해당 펀치가 포함된 드릴 추천
- 덜 연습한 콤보 우선 (콤보 노출 데이터 활용)
- LLM이 추천 이유를 한국어로 설명
- 세션 설정 화면에 추천 사항 자동 채움

**F7. 성과 히스토리**
- 시간 경과에 따른 펀치별 정확도 추이
- 세션 점수 히스토리
- 시각적 진행 지표

### P2: 추후 고려 (Post-MVP)

**F8. 폼 품질 분석**
- "훅을 던졌는가?" 너머 → "훅을 잘 던졌는가?"
- 훨씬 더 복잡한 ML 필요

**F9. 어노테이션 포함 영상 재생**
- 감지된 펀치가 하이라이트된 세션 영상 재생
- 실수한 타임스탬프 표시

## 7. 사용자 시나리오

### US1: 첫 영상 분석
> 드릴 세션을 마친 사용자로서, 훈련 영상을 업로드하고 점수를 받아서 드릴을 얼마나 잘 따라했는지 보고 싶다.

**플로우:**
1. 오디오 드릴 세션 완료 (기존 플로우)
2. 세션 종료 화면에 "영상 분석 업로드" 버튼 표시
3. 사용자가 카메라 롤에서 영상 선택
4. 업로드 시작, 진행률 표시
5. 처리 화면에 상태 업데이트 (업로드 중 → 분석 중 → 완료)
6. 결과 화면에 점수, 세그먼트별 분석, 코칭 피드백 표시

### US2: 코칭 피드백
> 세션에서 65%를 받은 사용자로서, 구체적으로 뭘 틀렸는지 알고 싶다.

**피드백 예시:**
> "원투 훅 콤비네이션에서 훅이 자주 빠졌어요. 8개 세그먼트 중 3개에서 훅을 놓쳤습니다. 다음 세션에서는 훅을 의식적으로 마무리해보세요."

### US3: 적응형 다음 세션
> 3세션 연속 훅 정확도가 낮은 사용자로서, 앱이 자동으로 현재 레벨에 맞는 훅 중심 드릴을 제안해주길 원한다.

**플로우:**
1. 분석 완료 후, 피드백 아래에 추천 표시
2. "추천 다음 세션: 중급 / 훅 포커스" + 설명 문구
3. "추천 세션 시작" 탭 → 세션 설정 자동 채움
4. 무시하고 수동 선택도 가능

### US4: 성장 추적
> 2주간 훈련한 사용자로서, 잽 정확도가 70%에서 90%로 올랐다는 걸 보고 동기 부여를 받고 싶다.

## 8. 범위

### 포함
- 세션 후 영상 분석 (훈련 후 업로드)
- 잽, 크로스, 훅, 어퍼컷, 바디샷 인식 (5가지 펀치 + idle)
- 드릴 플랜 대비 세그먼트별 비교
- LLM 기반 한국어 코칭 피드백
- 규칙 기반 다음 세션 추천
- 모바일 UI (업로드, 상태, 결과, 추천)

### 제외
- 세션 중 실시간 영상 처리
- 폼/기술 품질 분석 ("어떤 펀치"만 확인, "얼마나 잘"은 안 함)
- 방어 동작 감지 (슬립, 덕킹, 위빙 — 채점에서 필터)
- 추가 펀치 타입 (엘보, 킥 등 비복싱 동작)
- 다중 사용자 지원 (싱글 유저 MVP)
- 클라우드 영상 저장 (로컬 파일시스템)
- 디바이스 영상 압축

### 제약 조건
- **싱글 유저 MVP:** 인증, 멀티테넌시 없음
- **비동기 처리:** 영상 분석은 백그라운드 실행, 모바일에서 폴링
- **비용 통제:** 분석당 LLM 최대 2회 호출 (피드백 + 추천 내러티브)
- **로컬 저장소:** 영상은 서버 파일시스템(data/videos/)에 저장

## 9. 주요 리스크

| 리스크 | 영향 | 가능성 | 완화 방안 |
|--------|------|--------|-----------|
| AlphaPose(학습) vs MediaPipe(추론) 데이터 불일치로 TCN 정확도 하락 | 높음 — 핵심 기능 무력화 | 중간 | 공유 특성 공학 레이어로 양쪽 소스를 동일한 24차원 표현으로 정규화; 교차 소스 테스트셋으로 검증 |
| BoxingVI 데이터셋 라벨이 6개 클래스(idle+5펀치)로 깔끔하게 매핑 안 됨 | 높음 — ML 학습 차단 | 중간 | 검증 게이트: 학습 파이프라인 커밋 전 데이터셋 먼저 점검 |
| 5분 처리 목표 미달성 | 중간 — UX 저하 | 낮음 | 10fps 샘플링 + 휴식 구간 스킵으로 프레임 ~70% 감소; 해상도 낮추기 폴백 |
| 오디오-영상 싱크 실패 (영상 시작/종료가 오디오와 불일치) | 중간 — 잘못된 세그먼트 비교 | 중간 | MVP에서 첫 감지 펀치를 앵커로 사용; 세그먼트당 ±1초 버퍼 |
| LLM 피드백이 뭉뚱그리거나 도움 안 됨 | 중간 — 코칭 가치 없음 | 낮음 | 프롬프트에 세그먼트별 데이터(콤보명, 점수, 놓친 펀치) 포함; 3개+ 시나리오 품질 검토 |

## 10. 기술 접근 (상위 수준)

### ML 파이프라인
```
영상 → MediaPipe(포즈) → 특성 공학(24차원) → TCN(6클래스) → 이벤트 추출 → 비교
```

- **포즈 추정:** MediaPipe (IMAGE 모드, 10fps 샘플링)
- **특성 공학:** 6개 관절 부분집합 (어깨, 팔꿈치, 손목), 엉덩이 중심 정규화, 속도 특성 → 프레임당 24차원 벡터. 공유 모듈이 AlphaPose(학습)와 MediaPipe(추론) 모두 처리
- **동작 인식:** 3층 dilated TCN (Conv1D, dilation 1/2/4) → 6클래스 프레임 레벨 분류 (idle + 잽, 크로스, 훅, 어퍼컷, 바디샷)
- **학습 데이터:** BoxingVI 데이터셋 (AlphaPose 키포인트)
- **이벤트 추출:** 연속 프레임 라벨 그룹화, 최소 지속 시간 필터

### 비교 엔진
- 세션 전체가 아닌 세그먼트별 비교 (드릴 플랜 타임스탬프 활용)
- 세그먼트별 편집 거리, 세션 점수로 집계
- 방어 동작은 필터 (페널티 없음)

### 시스템 통합
- 기존 FastAPI 백엔드, SQLite DB, LLMClient 위에 구축
- 새 DB 테이블 3개 (video_uploads, analysis_results, session_recommendations)
- 새 API 라우터 (/api/analysis) — 업로드, 상태, 결과, 추천 엔드포인트
- 모바일: 영상 업로드, 처리 상태, 피드백 표시 새 화면

## 11. 단계별 계획

### Phase 1: ML 기반
- 공유 특성 공학 모듈 (AlphaPose + MediaPipe 호환)
- BoxingVI 데이터셋 검증 및 준비
- TCN 모델 학습 및 평가
- **게이트:** 클래스별 F1 > 0.7 달성 후 다음 단계 진행

### Phase 2: 서버 파이프라인
- 데이터 모델 + 마이그레이션
- 영상 업로드 엔드포인트 + 저장
- 포즈 추출 + TCN 추론 서비스
- 분석 오케스트레이터 (전체 파이프라인 조율)
- 시퀀스 비교 엔진
- LLM 피드백 생성

### Phase 3: 사용자 대면
- 모바일 영상 촬영/선택
- 업로드 + 진행률
- 처리 상태 폴링
- 피드백 + 추천 표시
- 추천으로 세션 설정 자동 채움

### Phase 4: 피드백 루프 (Post-MVP)
- 성과 히스토리 및 추이
- 추가 동작 타입 확장 (비복싱 동작 등)
- 폼 품질 힌트

## 12. 현재 시스템 의존성

| 현재 기능 | 의존 관계 | 영향 |
|-----------|-----------|------|
| 드릴 플랜 (plan_json) | 세그먼트 타임스탬프와 콤보명 포함 | 기대 시퀀스 도출 및 영상 시간 구간 분할에 사용 |
| combo_dictionary.json | 콤보명 → 액션 시퀀스 매핑 | 드릴 플랜 세그먼트를 기대 펀치 시퀀스로 변환 |
| SessionLog | 영상 업로드를 완료된 세션에 연결 | video_uploads.session_log_id FK |
| UserProfile | performance_summary_json 저장 | 추천 엔진이 업데이트 |
| LLMClient | 코칭 피드백 및 추천 내러티브 생성 | 기존 그대로 재사용 (분석당 1~2회 호출) |

**핵심 원칙:** 이 기능은 순수하게 추가적이다. 기존 드릴 세션 플로우는 수정하지 않는다.
